# Coding Agent History Viewer 开发文档

## 1. 产品边界

本项目是 Claude Code 与 OpenAI Codex 的本地历史查看器，技术栈为 Tauri v2、React/TypeScript 和 Rust。核心约束：

1. Claude Code 既有行为、数据和设置必须继续兼容。
2. Codex 必须接入项目导航、Prompt、最近记录、搜索、会话详情、工具调用、导出和所有统计，不是只接首页数字。
3. 所有历史只在本机只读处理。不得读取 Codex `auth.json`，不得把私有 SQLite 表作为主要数据源。
4. 两个来源可以独立缺失；另一个来源仍应完整可用。
5. Codex 历史可能超过 1 GB，JSONL 必须逐行流式解析，不能整文件读入内存。

```text

  codex_parser.rs        Codex adapter
  models.rs              公共领域模型与设置 DTO
  indexer.rs             并行扫描、归一化、合并、统计、cache v5
  pricing.rs             分产品/模型价格匹配与估算
  export.rs              Prompt/搜索/会话 Markdown
  commands.rs            Tauri command 边界
  state.rs               设置、路径解析和内存索引状态

src-tauri/tests/

  golden.rs              Claude adapter golden 验证
  codex_golden.rs        Codex adapter、索引与 cache golden 验证

  real_data_smoke.rs     显式 opt-in 的本机只读聚合冒烟
```

Claude 与 Codex adapter 分别理解各自上游格式，只输出统一的中间记录。索引层不应根据任一产品的原始 JSON 字段做分支解析。

```text
Claude files -> Claude adapter --+
                                  +-> normalized records -> index/cache -> commands -> React
Codex files  -> Codex adapter  ---+
```

## 3. 路径与设置

### 3.1 Claude Code

默认根目录是 `~/.claude`：

```text
history.jsonl
projects/**/*.jsonl
sessions/*.json
```

兼容旧设置字段：

```json
{
  "claudeDataDir": "",
  "historyFile": "",
  "projectsDir": "",
  "sessionsDir": ""
}
```

三个单项路径优先于 `claudeDataDir`，未设置时从 `~/.claude` 推导。

### 3.2 OpenAI Codex

Codex 根目录解析优先级必须固定为：

1. `settings.json` 的 `codexDataDir`
2. `CODEX_HOME`
3. `~/.codex`

读取范围：

```text
history.jsonl
sessions/YYYY/MM/DD/rollout-*.jsonl
archived_sessions/*.jsonl
```

`archived_sessions` 扫描所有 `.jsonl`，不能依赖文件名一定以 `rollout-` 开头。目录不存在不是错误。

### 3.3 设置迁移与解析状态

Serde 设置结构使用默认值反序列化。旧版文件没有 `codexDataDir` 时应得到空串，并按 `CODEX_HOME` 或 `~/.codex` 解析，不要求用户手工迁移。

设置 command 返回：

- 原始 Claude/Codex 配置值。
- 实际配置文件位置。
- Claude history/projects/sessions 最终绝对路径与存在状态。
- Codex history/sessions/archived 最终绝对路径与存在状态。

正式设置仍写入 bundle identifier `com.xzl.cchistoryviewer` 对应的应用配置目录。不得无必要更换 identifier；源码根 `settings.json` 只作为旧开发环境回退。

## 4. 原始格式解析

### 4.1 Claude adapter

Claude adapter 继续读取：

- `history.jsonl` 的 `display`、`timestamp`、`project`、`sessionId` 和粘贴计数。
- `projects/**/*.jsonl` 的 user/assistant/system 行、内容块、工具、分支、版本和 usage。
- `sessions/*.json` 的 CLI 版本补充信息。

继续保留斜杠命令清理、system reminder 清理、tool-result-only 排除、sidechain Prompt 排除、图片占位符归一化和 assistant usage 去重行为。

### 4.2 Codex history

`history.jsonl` 每行的当前形状：

```json
{"session_id":"synthetic-session","text":"Synthetic request","ts":1784275200}
```

`ts` 可能是 Unix 秒或毫秒。归一化规则应按数量级判断并输出毫秒，不能把秒误当作 1970 年附近的毫秒。

history 本身没有 `cwd`。索引通过 `session_id` 关联 rollout 的 `session_meta.id`，再使用 `session_meta.cwd` 归属项目。找不到 rollout 时可保留可安全表示的记录，但不能伪造真实目录。

### 4.3 Codex rollout

| 顶层 `type` | 读取内容 |
|---|---|
| `session_meta` | `id`、`cwd`、`cli_version`、`source`、时间 |
| `turn_context` | 当前 turn 的 `model` 与 `cwd` |
| `event_msg` / `user_message` | 当前格式真实用户 Prompt |
| `event_msg` / `token_count` | `info.last_token_usage` |
| `response_item` / `message` | 助手消息；旧格式 user fallback |
| `response_item` / function/custom tool | 工具调用及结果 |

未知顶层或 payload 类型直接忽略。单行 JSON 解析失败只跳过这一行，后续行继续处理。旧 rollout 可以没有 `event_msg`、`turn_context` 或 `token_count`。

### 4.4 Prompt 优先级与注入过滤

Codex Prompt 来源按以下规则处理：

1. `history.jsonl` 与 `event_msg.user_message` 都是真实候选，通过 session/cwd/文本/时间在 Codex agent 内合并。
2. 完全旧版 rollout 缺少真正的 `event_msg.user_message` 时，允许 `response_item role=user` 回退。
3. rollout 中途从旧格式切换到当前格式时，保留首个 event Prompt 之前互不重复的旧 Prompt；从首个 event 起只使用 `event_msg.user_message`，并在五分钟窗口内去掉切换点的同文本副本。
4. 回退必须排除 developer、system、`AGENTS.md`、`environment_context`、`codex_internal_context` 和类似运行时注入。
5. 若一条旧 user 内容同时含真实文本与可识别注入块，只保留清理后的真实文本；清理后为空则丢弃。

### 4.5 sub-agent

Codex `session_meta.source` 可能是字符串，也可能是带 `subagent` 的对象。默认：

- 自动生成的 sub-agent Prompt 不进入 Prompt 列表和 Prompt 统计。
- sub-agent 会话不进入默认会话统计。
- sub-agent 每次真实 `token_count` 仍进入用量统计，模型来自其 `turn_context`，项目来自其 `cwd`。

Claude 的 `projects/**/subagents/*.jsonl` 同样标记为 sub-agent：Prompt 与会话摘要隐藏，assistant usage 继续计入，详情消息保留 sidechain 标识。

## 5. 流式与容错要求

history 和 rollout 使用 `BufReader` 按行读取，或使用等价的有界缓冲实现。禁止对 Codex JSONL 使用 `read_to_string`/`read_to_end`。解析过程中只保留索引所需的会话摘要、Prompt、消息定位和 usage 事件，不能在建立索引时把全部原始 JSON 保存在内存。

文件列表可以通过 Rayon 并行解析，但每个文件内部保持事件顺序，因为模型切换与 token 归属依赖最近的 `turn_context`。

容错规则：

- 文件不存在、目录不存在：返回空来源。
- 损坏行、未知事件、缺少可选字段：跳过或降级，不能使文件或全局索引失败。
- 无 `event_msg` 的旧格式：使用受限 fallback。
- 无 Token/模型：会话仍可展示，统计相应字段为空。
- 超长内容块在展示模型中有界截断，但工具或图片大字段不能阻塞 Prompt 摘要解析。

## 6. 公共领域模型

核心枚举：

```rust
enum Agent { Claude, Codex }
enum AgentFilter { All, Claude, Codex }
enum PromptOrigin { History, Conversation, Both }
```

序列化为小写字符串。公共记录至少满足：

```text
PromptEntry:         id, agent, origin, text, project/cwd, timestamp, sessionId, ...
ProjectInfo:         path, agents, per-filter prompt/session counts, activity range, ...
SessionSummary:      agent, sessionId, project, title, time range, model/version, ...
ConversationDetail: agent, sessionId, project, messages, metadata, ...
ChatMessage:         agent, role, timestamp, blocks, ...
UsageEvent:          agent, model, project, timestamp, normalized usage, fingerprint
```

`origin` 仅表示 Prompt 在 `history`、`conversation` 或两者中出现，不能再用 `source` 同时表达产品。会话上的 `source` 字段仅对应 Codex `session_meta.source` 或 Claude CLI 来源。

### 6.1 ID、路由和去重

- Prompt 稳定 ID 的输入必须包含 `agent`、cwd、归一化时间和文本。
- 会话映射与路由键是 `(agent, session_id)`，不能只用 `session_id`。
- Prompt 只在同一 agent 内合并；跨产品的相同 cwd/时间/文本必须保留两条。
- 缓存记录与缓存 key 包含 agent。
- Token 事件指纹在同一 agent 内跨文件稳定，但不能让 Claude/Codex 互相去重。

### 6.2 项目合并

“全部”模式按 cwd 字符串集合合并项目。项目条目记录实际包含的 agents；Prompt 和会话本身始终保留 agent。统计恒等式：

```text
all.prompts = claude.prompts + codex.prompts
all.sessions = claude.sessions + codex.sessions
all.<each normalized token field> = claude + codex
all.projects = |set(claude.cwd) union set(codex.cwd)|
```

## 7. Token 与成本

### 7.1 统一用量

```rust
struct NormalizedUsage {
    uncached_input: u64,
    cache_read: u64,
    cache_creation: u64,
    output: u64,
    reasoning_output: u64,
}
```

派生总量：

```text
totalTokensIncludingCache = uncachedInput + cacheRead + cacheCreation + output
```

`reasoningOutput` 是 `output` 的子集，只供拆分展示，不加进总量。

Claude 映射：

```text
uncachedInput <- input_tokens
cacheRead <- cache_read_input_tokens
cacheCreation <- cache_creation_input_tokens
output <- output_tokens
```

Codex 每个 `token_count` 只读取 `info.last_token_usage`：

```text
uncachedInput <- input_tokens - cached_input_tokens
cacheRead <- cached_input_tokens
cacheCreation <- 0
output <- output_tokens
reasoningOutput <- reasoning_output_tokens
```

减法应防止畸形上游数据下溢。累计的 `info.total_token_usage` 绝不加入统计。

### 7.2 fork/resume 去重

派生会话会把旧事件复制到新文件。Token 指纹不能依赖：

- rollout 文件绝对路径
- 新 session ID

指纹应由可稳定识别同一原始调用的事件内容与上下文构成，并包含 agent。测试必须同时证明：

1. 不同文件/新 session 中复制的事件只计一次。
2. 数值完全相同、但实际是另一时刻/turn 的调用仍计一次新的用量。

### 7.3 缓存命中率

```text
cacheHitRate = cacheRead / (uncachedInput + cacheRead)
```

分母为 0 时领域值应为 `None/null`，前端显示“—”，不能显示 `0%` 误导用户。

### 7.4 成本

价格按 agent 与可靠匹配的完整模型/官方 snapshot 计算。未知模型返回 `None`，不能返回伪造的 0 美元。当前内置表的核对日期是 2026-07-17，依据 [Anthropic 定价](https://platform.claude.com/docs/en/about-claude/pricing) 和 OpenAI 官方模型页（例如 [GPT-5.5](https://developers.openai.com/api/docs/models/gpt-5.5)、[GPT-5.4](https://developers.openai.com/api/docs/models/gpt-5.4)、[GPT-5.4 mini](https://developers.openai.com/api/docs/models/gpt-5.4-mini)、[GPT-5.1 Codex Max](https://developers.openai.com/api/docs/models/gpt-5.1-codex-max)、[o4-mini](https://developers.openai.com/api/docs/models/o4-mini) 与 [GPT-4.1](https://developers.openai.com/api/docs/models/gpt-4.1)）；发布时必须重新核验可能变化的费率。

合并成本只累加已知模型，并同时返回 `unknownPriceTokens` 或等价覆盖字段。未知 Token 计数使用 `totalTokensIncludingCache` 的口径，不重复 reasoning。

Codex UI 和导出统一写“API 等价估算 / API-equivalent estimate”。它不是 ChatGPT/Codex 订阅实际支出、额度或积分。价格表变更必须附官方来源核对日期；无法可靠覆盖的模型保持未知。

## 8. 索引与 cache v5

缓存 schema 版本为 **5**。结构或解析规则变化时必须增加版本，旧 cache 自动失效。

v5 语义：

- Claude/Codex history 分别记录 agent、解析结果和文件 `mtime + length` 指纹。
- 每个 session/rollout 文件有 agent-aware 文件缓存，key 不能只依赖 session ID。
- 当前 Codex dated sessions 与 archived sessions 使用同一文件级增量策略。
- 启动时复用 `mtime + length` 指纹未变化的记录，并行解析新增或变化文件。
- 已删除文件从 cache 移除；数据根配置变化不能复用另一个根的陈旧记录。
- cache version、agent 或 parser 规则不符时重建。
- cache 写在本应用目录，可以含解析后的 Prompt/摘要/用量，因此属于本地敏感派生数据。

强制刷新忽略旧文件条目并完整重建，但仍不得修改源数据。cache 序列化宜使用 `BufWriter`/`serde_json::to_writer`，避免先生成一个巨大的 JSON 字符串。

## 9. Command 与前端契约

所有列表与统计 command 显式接收 `agentFilter`：

```text
get_projects(agentFilter)
get_project_prompts(project, sort, includeCommands, agentFilter)
get_recent_prompts(limit, includeCommands, agentFilter)
search_prompts(query, projectFilter, includeCommands, agentFilter)
get_stats(agentFilter)
get_project_sessions(project, agentFilter)
get_conversation(agent, sessionId)
build_prompt_export(..., agentFilter)
export_search_results(..., agentFilter)
export_conversation(agent, sessionId, ...)
```

具体参数命名以 Rust/TypeScript 同步定义为准，但以下要求不可省略：

- 会话详情和导出必须带精确 agent，避免相同 session ID 冲突。
- 首页筛选影响概览卡、活动图、小时/星期分布、项目排行、最近 Prompt、模型、CLI 版本、Token 和成本。
- 项目导航、文件夹 Prompt、全局/文件夹搜索也使用相同筛选状态。
- 筛选默认 `all`，通过 localStorage 持久化；无效旧值回退 `all`。
- React Query key 包含 filter/agent，切换时不得复用错误缓存。
- 导出全部来源时，每条或每个分组保留 agent 标签。

## 10. 导出

Prompt、搜索和会话导出继续支持中英文。通用标题不再写死 Claude Code。导出内容至少保留：

- agent
- cwd/项目
- 时间
- Prompt 或消息正文
- 会话 ID（适用时）
- 工具调用（用户选择包含时）

Codex 会话不能生成 `claude --resume` 命令；若提供继续命令，应按 agent 使用对应 CLI，或在无法保证语义时不展示。

只有 `write=true` 且用户主动执行导出时才写目标文件。预览不得触发磁盘写入。

## 11. 隐私与写入审计

允许读取：

- 上述 Claude/Codex JSON/JSONL 数据文件。

禁止读取或依赖：

- Codex `auth.json`
- Codex 私有 SQLite 表作为主数据源
- 无关的凭据、环境密钥或项目文件

允许写入：

- 应用配置目录的 `settings.json`
- 应用数据/配置目录的索引 cache
- 用户明确请求的导出文件

禁止修改、删除或在数据根下创建任何文件。应用不上传历史、不遥测 Prompt、不为价格或解析联网。

## 12. 测试策略

### 12.1 合成 fixture

`src-tauri/tests/fixtures/codex/` 只含虚构内容和 `/synthetic/...` 路径，覆盖：

- 当前 rollout 格式
- 旧 `response_item` fallback 与注入过滤
- 同一 rollout 从旧格式切换到 event 格式时的 Prompt 合并与切换点去重
- history/session 关联
- archived session
- 损坏行与未知事件
- 秒/毫秒 timestamp
- turn 内模型切换
- assistant/function/custom/tool-search/image-generation tool
- `last_token_usage` 与累计 total 的区别
- fork/resume 复制去重和相同数值的独立调用
- sub-agent Prompt 排除但 Token 保留

不得从真实 `~/.codex` 复制 fixture。

### 12.2 必须锁定的不变量

1. 两个 agent 同 session ID 可同时打开正确详情。
2. 跨 agent 相同文本不去重，稳定 ID 不同。
3. “全部”的 Prompt、会话和每个 Token 分项等于两者之和；项目数是 cwd 并集。
4. Claude cacheRead/cacheCreation 进入总 Token。
5. Codex cached input 不重复计算，reasoning 不重复计算。
6. 只累加 `last_token_usage`。
7. copied fork event 去重，distinct same-usage event 保留。
8. cache hit 分母为 0 时为空值。
9. Claude-only、Codex-only 和双源缺失均不崩溃。
10. cache 首次解析、完整命中、单文件变化、删除和版本失效符合 v5 语义。

### 12.3 发布门禁

```bash
cd src-tauri
cargo fmt --check
cargo test
cd ..
pnpm build
```

`pnpm build` 负责生产前端构建；若需要独立 TypeScript 门禁，执行 `pnpm exec tsc --noEmit`。

真实 Claude/Codex 自动冒烟只读加载两个来源，并验证源文件元数据不变、合并统计恒等式、项目路径并集和总 Token 公式。该测试不提供 cache 路径，也不打开会话详情或工具展示；cache 命中与详情 UI 另行通过合成测试和界面验收确认。报告只写结果，不输出 Prompt、私人绝对路径、凭据或 cache 内容。

## 13. 上游兼容风险

- Claude Code 和 Codex 的本地格式均未承诺永久稳定；未知字段必须宽容。
- Codex `source` 的字符串/对象形状、tool item 类型和注入包装可能继续增加。
- 模型别名与 API 价格会变化；严格匹配失败时显示未知比猜价安全。
- fork/resume 复制策略变化可能需要升级事件指纹和 cache schema。
- 文件 `mtime` 精度因平台不同；增量行为必须由 fixture/tempdir 测试锁定。
